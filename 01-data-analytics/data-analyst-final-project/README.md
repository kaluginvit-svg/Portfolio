# A/B-тест в финтехе: позиция карточки «Кредитка дня» в ленте

**Кейс для портфеля:** от сырых событий до продуктовой рекомендации — с честной оценкой мощности теста и без «магии p-value».

Я закрываю полный цикл: дизайн метрик под ТЗ, SQL/пайплайн на user-day → user-level, Welch t-test, расчёт выборки под MDE, QC датасета и формулировка вывода, который можно отнести в продакт без натяжки.

**Суть эксперимента:** поднять карточку `credit_of_day` выше в ленте и проверить влияние на `CR_apply` (primary), `CTR`, `CR_home_apply`, `ARPU`, Approval Rate. Схема данных: `dt | client_id | ab_group | event_type | offer_id | value`.

## Что внутри (коротко)

| Что сделано | Как |
|-------------|-----|
| Первичная метрика | `CR_apply = offer_apply / offer_impression` |
| Статистика | User-level Welch t-test |
| MDE по ТЗ | 2,5% → расчёт требуемой выборки |
| Вывод | Сильный **directional** uplift по `CR_apply` и `CTR` в данных; для заявленного MDE на 7-дневном окне **мощности недостаточно** — rollout «как доказанный факт» не подкреплён |

## Структура репозитория

`data/raw` (seed + расширенный CSV) → `data/processed` (funnel, метрики, `ab_results`, MDE, QC) → `sql/`, `src/`, `notebooks/ab_test_analysis_showcase.ipynb` → `reports/final_report.md`, `reports/presentation.html`, `reports/figures/`. Детали дизайна и метрик — в `docs/`.

Расширение датасета до 20k строк — **bootstrap целых пользовательских траекторий**, не построчный random; для демо пайплайна и воспроизводимости, не замена реального независимого трафика.

## Запуск

```bash
python -m venv .venv
# Windows: .venv\Scripts\activate
pip install -r requirements.txt
make all
```

Или по шагам: `make data` → `make analysis` → `make report`.

## Почему это про меня как аналитика

- Связываю **бизнес-вопрос** (видимость оффера → заявки) с **корректной единицей анализа** (пользователь, не сессия «ради значимости»).
- Не путаю **статистическую значимость** с **практической проверкой малого эффекта**: явно считаю мощность и размер выборки под MDE.
- Даю **рекомендацию с оговорками**: перспективно по сигналу, но нужны больший трафик или пересогласованный MDE / горизонт теста.

**Дальше по кейсу:** больше независимого трафика, сегменты (новые/старые, платформы), долгий горизонт по approval/revenue, проверка каннибализации других карточек.
