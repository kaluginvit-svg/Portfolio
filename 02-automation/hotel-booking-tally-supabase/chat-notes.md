## 1) ТЗ проекта (из `Итоговый_проект.txt` и блока в `5.html`)

### Цель
Автоматизировать обработку заявок на бронирование отелей без ручного переноса данных.

### Что нужно автоматизировать
- приём заявки через форму;
- проверка свободных номеров в таблице;
- создание бронирования;
- изменение статуса номера;
- отправка подтверждения клиенту;
- отчёт менеджеру по новым броням.

### Таблицы/данные
Нужно 2 таблицы:

**`Rooms`** (колонки):
- `room_id`
- `hotel_name`
- `room_type`
- `date_from`
- `date_to`
- `status` (минимум: `free`, `reserved`)
- `price`

**`Bookings`** (колонки):
- `booking_id`
- `client_name`
- `email`
- `hotel_name`
- `room_type`
- `date_from`
- `date_to`
- `status` (минимум: `reserved`; опционально `confirmed`, `cancelled`)
- `payment_status` (например `pending`, `paid`)
- **`created_at`** — добавляем (для отчёта “новые за сутки”)

### Форма заявки (поля)
- имя;
- почта;
- отель;
- тип номера;
- дата заезда;
- дата выезда.

### Поиск номера (условия)
- совпадает `hotel_name`;
- совпадает `room_type`;
- `status = free`;
- даты подходят.

**Трактовка дат “по смыслу ТЗ”**: `Rooms.date_from/date_to` — диапазон доступности комнаты.  
Условие “даты подходят”:
- `Rooms.date_from <= request.date_from`
- `Rooms.date_to >= request.date_to`

### Если номер найден
- создать запись в `Bookings` со статусом `reserved`;
- обновить номер в `Rooms`: `status = reserved`;
- отправить письмо клиенту (отель/даты/тип/стоимость/инструкция по оплате или подтверждению).

### Если номер не найден
Отправить письмо клиенту: свободных номеров нет, менеджер предложит альтернативы.

### Ежедневный отчёт менеджеру
Каждый день в **18:00** отправлять менеджеру:
- новые бронирования;
- неподтверждённые брони;
- номера со статусом `reserved`.

### Что сдавать (артефакты)
1. Скрин сценария обработки заявки.
2. Скрин таблиц Rooms и Bookings.
3. Скрин письма клиенту.
4. Скрин ежедневного отчёта менеджеру.
5. Скрин успешных запусков.

## 2) План реализации (архитектура n8n)

Рекомендуется сделать **2 воркфлоу**.

### Воркфлоу A — обработка заявки (Webhook от Tally)
**Цель**: принять заявку, проверить доступность, зарезервировать, создать бронь, отправить письма.

**Шаги (универсально):**
1) Trigger: **Webhook** (Tally) — получить payload.
2) Validate/Normalize (Set/Code):
   - привести даты к `YYYY-MM-DD`,
   - нормализовать `hotel_name`/`room_type`,
   - сформировать `request_id` (антидубли: например hash(email+hotel+room_type+date_from+date_to)).
3) Idempotency check:
   - проверить в `Bookings`, нет ли уже записи с этим `request_id`.
   - если есть — завершить без повторного создания.
4) Search free room:
   - запрос `Rooms` по `hotel_name`, `room_type`, `status=free` + проверка диапазона дат.
5) IF room found?
   - NO → письмо клиенту “нет свободных номеров” (+ опционально уведомить менеджера).
   - YES →
6) Atomic reserve (защита от гонок):
   - обновить `Rooms.status` на `reserved` **только если** он всё ещё `free`.
   - если обновление не прошло — попробовать следующий вариант (если реализовано) или уйти в ветку NO.
7) Create booking (`Bookings`):
   - `status=reserved`, `payment_status=pending`, `created_at=now()`,
   - сохранить `room_id`, `price` и параметры заявки.
8) Send email to client (подтверждение + инструкция).
9) Notify manager (опционально, сразу) — кратко по брони.
10) Logging — результат выполнения (Executions n8n и/или отдельная таблица логов).

### Воркфлоу B — ежедневный отчёт менеджеру (Cron 18:00)
1) Cron: ежедневно 18:00.
2) Query `Bookings`:
   - новые за сутки (по `created_at`),
   - неподтверждённые (например `payment_status=pending` или `status=reserved` дольше N часов/дней).
3) Query `Rooms`: `status=reserved`.
4) Compose report (HTML/Markdown).
5) Send email manager на `dir@kalugin-consulting.ru` (Gmail).

### 1. Название проекта

### 3. Краткое описание проекта (готовый текст)
Автоматизация обрабатывает заявки на бронирование отелей в туристическом агентстве. Клиент оставляет заявку через форму, сценарий проверяет наличие свободного номера в базе, создаёт бронь, меняет статус номера и отправляет клиенту письмо. Менеджер получает уведомления и ежедневный отчёт по броням.

### 4. Бизнес-проблема (под кейс)
- Раньше менеджер вручную проверял номера, переносил данные, менял статусы и писал клиенту.
- Риски: задержки, ошибки, потери заявок, забытые статусы, **двойные бронирования**.
- Автоматизация нужна для ускорения обработки и снижения ошибок.

### 5. Цель автоматизации (под кейс)
Успех: после отправки заявки система автоматически проверяет наличие номера, резервирует, создаёт бронь, отправляет письмо клиенту; при отсутствии — отправляет отказ. Менеджер получает ежедневную сводку без ручной сборки.

### 7. Вход/выход
**Вход**: Tally → Webhook; поля формы: `client_name`, `email`, `hotel_name`, `room_type`, `date_from`, `date_to`.  
**Выход**: записи в Supabase (`Bookings`), обновление `Rooms.status`, письма Gmail, ежедневный отчёт.

### 8. Используемые сервисы
- n8n — оркестрация процессов
- Supabase — база данных (Rooms/Bookings)
- Tally — форма
- Gmail — отправка писем
- (для схемы) любой сервис схем/досок (draw.io/ProcessOn/Miro/VK Доска и т.д.)

### 9. Схема процесса (что указать)
- 2 ветки: `roomFound` / `roomNotFound`.
- Контрольные точки: валидация обязательных полей, антидубли, условное резервирование (против гонок), логирование ошибок, ежедневный отчёт.

### 10. Реализация сценария в n8n
#### 10.1 Общая логика (готовая формулировка)
Создано 2 рабочих процесса: первый принимает заявку из формы и выполняет подбор/резерв/создание брони и отправку письма клиенту; второй по расписанию формирует ежедневный отчёт и отправляет менеджеру.

#### 10.2 Основные шаги сценария (готовая “таблица” на 6 шагов)
1) Получение заявки — Tally → Webhook (n8n)  
2) Валидация/нормализация — n8n  
3) Поиск свободного номера — Supabase (`Rooms`)  
4) Резервирование номера — Supabase (`Rooms.status: free→reserved`)  
5) Создание брони — Supabase (`Bookings`, +`created_at`)  
6) Уведомления — Gmail (клиент + отчёт менеджеру)

#### 10.3 Работа с данными (готовый текст)
- Передавались поля: имя, email, отель, тип, даты, найденный `room_id`, `price`, статусы.  
- Преобразования: нормализация дат, формирование текста писем.  
- Условия: развилка “номер найден / не найден”; фильтры по `hotel_name`, `room_type`, `status`, диапазону дат.  
- Проверка обязательных полей: в начале сценария; при ошибке — выполнение фиксируется, бронь не создаётся.

### 11. Нейросеть
Если нейросеть не используется: “Не использовалась”.

### 12. Хранение данных (готовая структура)
Хранение в Supabase. Созданы таблицы `Rooms` и `Bookings`, используются статусы `free/reserved` и `pending/paid`. Для “новые за сутки” используется `created_at` в `Bookings`. История и контроль — через записи в таблицах и журнал выполнений n8n.

### 13. Обработка ошибок (готовый текст)
- Неполные данные → сценарий не создаёт бронь, фиксирует ошибку, при необходимости уведомляет менеджера.  
- Номер не найден → письмо клиенту “нет номеров”.  
- Ошибка сервиса (Supabase/Gmail) → ошибка фиксируется; возможны повтор/уведомление.  
- Защита от дублей → проверка по `request_id` перед созданием брони.

### 14. Тестирование (шаблон 3–5 запусков)
1) Номер есть → бронь создана → письмо подтверждения.  
2) Номера нет → письмо отказа → без брони.  
3) Дубль заявки → повторная бронь не создаётся.  
4) Неполные данные → ошибка без брони.  
5) Отчёт 18:00 → письмо менеджеру.

### 19. Текст для портфолио (готовый текст)
Я разработал(а) автоматизацию для туристического агентства, которая ускоряет обработку заявок на бронирование отелей и снижает риск ошибок. Раньше менеджер вручную проверял доступные номера, переносил данные в таблицы, менял статусы и отдельно писал клиенту, из‑за чего заявки обрабатывались медленно и возникали риски двойного бронирования.

В решении на n8n клиент заполняет форму (Tally), после чего сценарий автоматически валидирует данные, проверяет наличие свободного номера в базе Supabase, резервирует номер, создаёт запись брони и отправляет клиенту письмо с подтверждением и инструкцией. Если подходящего номера нет — клиент получает уведомление, что менеджер предложит альтернативы.

Дополнительно система формирует ежедневный отчёт в 18:00 и отправляет руководителю сводку по новым и неподтверждённым бронированиям, а также по номерам со статусом reserved. В результате процесс стал прозрачным и предсказуемым: данные хранятся в единой базе, статусы обновляются автоматически, а клиент быстрее получает ответ.
