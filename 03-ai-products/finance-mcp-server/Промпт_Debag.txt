========================================
4. КЛЮЧЕВЫЕ ПРОВЕРКИ
========================================

Нужно автоматически покрыть следующие типы проверок.

----------------------------------------
A. SMOKE TESTS
----------------------------------------

1. server startup
Проверить, что:
- проект стартует без критической ошибки
- создается БД
- создаются таблицы
- seed-данные загружаются
- доступны ключевые tools

2. health_check
Проверить:
- возвращается статус
- есть путь к БД
- есть counts по таблицам
- counts не пустые после seed

3. list_companies
Проверить:
- возвращается непустой список компаний

----------------------------------------
B. TOOL REGISTRY TESTS
----------------------------------------

Проверить:
- все ключевые tools зарегистрированы
- у каждого есть name
- description
- input schema
- output schema или предсказуемый JSON response
- registry не содержит дубликатов имен

Минимальный набор tools для проверки:
- health_check
- list_companies
- import_csv
- import_contract
- list_financial_records
- list_budget_records
- list_cash_positions
- list_contracts
- calculate_kpis
- plan_vs_fact
- liquidity_forecast
- payment_calendar
- contract_risk_scan
- add_investment_project
- evaluate_investment
- export_report
- calculate

----------------------------------------
C. IMPORT TESTS
----------------------------------------

1. import valid P&L CSV
Ожидание:
- imported_count > 0
- rejected_count == 0 или минимально
- записи появляются в financial_records

2. import valid budget CSV
Ожидание:
- записи появляются в budget_records

3. import payments CSV
Ожидание:
- записи появляются и доступны для payment_calendar / liquidity_forecast

4. import invalid CSV
Ожидание:
- сервер не падает
- возвращается structured error
- imported_count == 0 или частичный импорт
- errors содержит понятные сообщения

5. import contract from TXT
Ожидание:
- contract created
- raw_text сохранен
- часть полей распознана

6. import contract missing fields
Ожидание:
- создается запись
- warnings заполнены
- risk scan потом отлавливает missing terms

----------------------------------------
D. FINANCIAL CALCULATION TESTS
----------------------------------------

1. calculate_kpis
Проверить:
- ответ содержит все ожидаемые поля
- поля числовые там, где должны быть числа
- EBITDA присутствует
- cash_balance присутствует
- response format стабилен

2. plan_vs_fact
Проверить:
- total_plan
- total_fact
- variance_abs
- variance_pct
- breakdown_by_category
- данные не пустые

3. liquidity_forecast
Проверить:
- opening_cash
- projected_inflows
- projected_outflows
- ending_cash
- daily_projection
- risk_flags
- daily_projection не пустой

4. payment_calendar
Проверить:
- записи возвращаются
- есть direction
- есть overdue

----------------------------------------
E. CONTRACT TESTS
----------------------------------------

1. list_contracts
Проверить:
- contracts доступны
- active_only фильтр работает, если он реализован

2. contract_risk_scan
Проверить:
- на тестовых контрактах находятся риски
- risks сохраняются в alerts, если логика это поддерживает
- response содержит severity / message / related contract

----------------------------------------
F. INVESTMENT TESTS
----------------------------------------

1. add_investment_project
Проверить:
- проект добавляется
- запись появляется в БД

2. evaluate_investment
Проверить:
- есть NPV
- есть IRR
- есть payback_period
- есть profitability_index
- есть recommendation

----------------------------------------
G. SAFE CALCULATOR TESTS
----------------------------------------

1. valid math expression
Например:
- 2 + 2
- 10 / 2
- round(10.567, 2)
- max(10, 20, 5)

2. invalid / unsafe expression
Например:
- __import__("os")
- open("file.txt")
- lambda x: x
- class A: pass

Ожидание:
- unsafe expression blocked
- сервер не падает
- возвращается controlled error

----------------------------------------
H. EXPORT TESTS
----------------------------------------

Проверить export_report для типов:
- kpis
- plan_vs_fact
- liquidity_forecast
- payment_calendar
- contract_risks
- investment_evaluation

Ожидание:
- файл создается
- путь возвращается
- формат валиден
- export directory существует

----------------------------------------
I. NEGATIVE TESTS
----------------------------------------

Проверить:
- неизвестный statement_type
- отсутствующий file_path
- пустой input file
- некорректная дата
- несуществующая company_name
- evaluate_investment с несуществующим project_id
- export_report с неизвестным report_type

Ожидание:
- controlled failure
- понятная ошибка
- отсутствие crash

5. TEST HARNESS / SCENARIO RUNNER
========================================

Создай отдельный сценарный раннер:

Назначение:

- запускать сценарии последовательно
- фиксировать pass/fail
- сохранять traceback, если есть ошибка
- сохранять input/output кратко
- показывать summary в конце

Формат JSON отчета:
{
  "timestamp": "...",
  "project": "product-mcp",
  "summary": {
    "total": 0,
    "passed": 0,
    "failed": 0
  },
  "scenarios": [
    {
      "name": "startup_and_seed",
      "status": "passed",
      "details": "...",
      "error": null
    }
  ]
}
