# Аудит `bad_code.py`

Полный разбор исходной версии файла: найденные проблемы, риски и соответствие исправлений после рефакторинга.

---

## 1. Критические и высокие риски

| # | Проблема | Последствия | Исправление |
|---|----------|-------------|-------------|
| 1 | **SQL-инъекция**: строка запроса собиралась через f-string с `name` и `tags` | Выполнение произвольного SQL при вредоносном вводе | Параметризованный запрос: `execute("... VALUES (?, ?)", (name, payload))` |
| 2 | **«Хеш» пароля** как `password[::-1]` | Пароль восстанавливается одной операцией; нет соли и стойкости | `hashlib.pbkdf2_hmac` + случайная соль (`secrets.token_bytes`), запись в файл в структурированном виде |
| 3 | Соединение **SQLite без гарантированного закрытия** при исключении | Утечки дескрипторов, блокировки БД | Контекстный менеджер `with sqlite3.connect(...)` |
| 4 | Файл паролей открывался без **гарантии закрытия** при ошибке между `open` и концом | Утечки дескрипторов, частичная запись | `with Path(...).open("a", encoding="utf-8")` |

## 2. Логика и типы

| # | Проблема | Детали | Исправление |
|---|----------|--------|-------------|
| 5 | **Неверный тип возврата** | Докстринг обещал `int`, фактически `return cur.lastrowid + ""` давал **строку** | Явный `return int(last_id)` после проверки на `None` |
| 6 | **Мутируемый аргумент по умолчанию** `tags=[]` | Один общий список на все вызовы; «залипание» данных между вызовами | `tags: Optional[list] = None` и внутри `if tags is None: tags = []` |
| 7 | **Побочный эффект на аргументе** `tags.append("new")` | Вызывающий код не ожидает изменения переданного списка | Работа с копией: `row_tags = list(tags); row_tags.append("new")` |

## 3. Структура и сопровождение

| # | Проблема | Детали | Исправление |
|---|----------|--------|-------------|
| 8 | Неиспользуемая **глобальная переменная** `CACHE` | Шум, риск случайного использования как общего состояния | Удалена |
| 9 | Неиспользуемые импорты **`threading`**, **`time`** | Засорение модуля | Удалены |
| 10 | Синтаксическая/редакторная **ошибка в конце файла** | Строка `</code>` — явный мусор (вероятно HTML), ломает читаемость и мог бы сломать парсер | Удалена; файл приведён к валидному Python |
| 11 | Отсутствие **кодировки** при записи паролей | На Windows возможны проблемы с не-ASCII путями/содержимым | Явное `encoding="utf-8"` при открытии файла |

## 4. Что нужно сделать вручную при интеграции

- **Схема БД**: код предполагает таблицу `users` с колонками `name`, `tags` (тип текста/JSON по вашему выбору). При необходимости добавьте `CREATE TABLE IF NOT EXISTS ...`.
- **Формат `passwords.txt`**: после рефакторинга строка имеет вид  
  `user_id:salt_hex:iterations:dk_hex` — старые строки в формате `id:reversed_password` несовместимы; при миграции нужна отдельная политика (сброс паролей или одноразовый скрипт).
- **Проверка пароля при логине**: в модуле есть только **сохранение**; для входа нужно читать соль и итерации из файла/БД и сравнивать через `hmac.compare_digest` (константное время).

## 5. Краткое резюме

Исходный файл демонстрировал типичные антипаттерны: инъекции, слабое хранение секретов, мутабельные дефолты, несоответствие типов возврата и небезопасное управление ресурсами. Текущая версия устраняет перечисленное и задаёт основу для безопасного хранения паролей; дальнейшая доработка — миграция хранилища паролей в БД и единая схема таблиц.

---

## 6. Выделение публичного API и слоёв

**Да, публичный API лучше выделить отдельно**: так UI, CLI или HTTP-обработчики зависят только от стабильных имён (`add_user`, `store_password`), а детали SQLite и формата файла паролей можно менять в `users_repository` / `password_storage` без правок «внешних» вызовов.

Текущая раскладка:

| Модуль | Роль |
|--------|------|
| `api.py` | **Публичный API** — `__all__`, функции для внешнего кода и будущих маршрутов. |
| `users_repository.py` | Вставка пользователя в БД (`insert_user`). |
| `password_storage.py` | Запись PBKDF2 в файл (`append_password_hash`). |
| `settings.py` | Пути и параметры (`DB_PATH`, `PASSWORDS_FILE`, итерации PBKDF2). |
| `bad_code.py` | Тонкая прослойка совместимости: реэкспорт с `api` для старых импортов `from bad_code import ...`. |

**HTTP API** (FastAPI, Flask и т.д.) — отдельный слой: маршруты только вызывают функции из `api.py`, не обращаются к репозиториям напрямую. Это упростит тесты (мокаете `api`) и смену фреймворка.
