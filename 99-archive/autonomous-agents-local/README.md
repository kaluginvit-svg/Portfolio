# Псевдоагент: разработка SEO-ядра

Микросервис для генерации SEO-ядра по ссылке на сайт с помощью ChatGPT.

## Архитектура

- **Бэкенд**: FastAPI + OpenAI (модуль `openai`) — папка `backend/`
- **Фронтенд**: отдельная папка `frontend/` (HTML, CSS, JS); отдаётся бэкендом через Jinja2 и StaticFiles
- **Контейнеризация**: Docker Compose (фронтенд монтируется в контейнер бэкенда)

## Запуск

1. Скопируйте конфиг и задайте ключ OpenAI:
   ```bash
   copy .env.example .env
   ```
   В `.env` укажите `OPENAI_API_KEY=sk-...`.

2. Соберите и запустите:
   ```bash
   docker compose -p seo-agent up -d --build
   ```
   Или запустите скрипт из корня проекта:
   - **CMD:** `start.bat`
   - **PowerShell:** `.\start.bat` (обязательно с `.\`)

3. Откройте в браузере: http://localhost:8000

## Функционал

- На вход принимается **список URL** — вставьте ссылки в поле (каждый с новой строки или через запятую). Бэкенд анализирует каждый сайт и формирует **одно совокупное SEO-ядро**, чтобы опередить все сайты из списка.

Общий поток:
1. Пользователь вставляет список URL и нажимает «Анализировать».
2. Бэкенд запрашивает у ChatGPT план из 5–6 шагов (JSON с промптами).
3. Для каждого URL: загружается HTML (если не `USE_URL_DIRECTLY`), выполняются шаги плана, ответы собираются.
4. При одном URL — итоговое SEO-ядро по нему. При нескольких — единое SEO-ядро, объединяющее сильные стороны и перекрывающее слабые места всех конкурентов.
5. Лог и результат отображаются в реальном времени (SSE).

## Переменные окружения

| Переменная           | Описание |
|----------------------|----------|
| `OPENAI_API_KEY`     | Ключ API (OpenAI или ProxyAPI, обязательно) |
| `OPENAI_BASE_URL`    | URL прокси-API (если используете ProxyAPI вместо прямого OpenAI) |
| `OPENAI_SSL_VERIFY`  | Проверка SSL (по умолчанию `true`; при ошибках соединения с прокси можно попробовать `false`) |
| `OPENAI_MODEL`       | Модель (по умолчанию `gpt-4o-mini`) |
| `USE_URL_DIRECTLY`   | `true` — передавать в промпты только URL; `false` — подставлять загруженный HTML |

## Устранение неполадок

**Ошибка «Not Found» (404)**  
- Открывайте приложение по адресу http://localhost:8000 (не с диска через `file://`).  
- Проверьте, что бэкенд запущен: откройте http://localhost:8000/api/status — должна открыться страница с конфигом.

**Анализ не запускается (ключ и баланс ProxyAPI в порядке)**  
Сделайте по шагам:

1. **Пересоздать контейнер** (чтобы подхватить `.env`):
   ```bash
   docker compose -p seo-agent down
   docker compose -p seo-agent up -d
   ```
2. **Проверить конфиг в контейнере** — откройте в браузере: http://localhost:8000/api/status  
   Должно быть: `openai_api_key_set: true`, `openai_base_url` — URL вашего ProxyAPI (не «не задан», если используете прокси).
3. **Проверить `.env`** — файл в **корне проекта** (рядом с `docker-compose.yml`). Обязательно: `OPENAI_API_KEY=...` и `OPENAI_BASE_URL=https://...` (URL из документации ProxyAPI).
4. **Логи бэкенда** при нажатии «Анализировать»:
   ```bash
   docker logs seo-agent-backend -f
   ```
5. **Браузер** — F12 → Network: идёт ли запрос к POST `/api/analyze` и какой ответ.

**`TLS handshake timeout` / не удаётся скачать образ `python:3.12-slim`**  
Таймаут при обращении к Docker Hub (registry-1.docker.io). Что сделать:

1. **Повторить сборку** — иногда срабатывает при повторном запуске.
2. **Проверить сеть** — VPN, фаервол, антивирус; при необходимости отключить/включить VPN и попробовать снова.
3. **Скачать образ отдельно** (удобно при нестабильном канале):
   ```bash
   docker pull python:3.12-slim
   docker compose -p seo-agent up -d --build
   ```
4. **Зеркало Docker Hub** — если есть корпоративное или региональное зеркало, укажите его в настройках Docker (например, в Desktop: Settings → Docker Engine → `"registry-mirrors": ["https://..."]`).

**`Ошибка соединения` / `Connection error` при запросе к ProxyAPI**  
Запрос до API не доходит или прокси обрывает соединение. Проверьте по порядку:

1. **Баланс ProxyAPI** — часто при пустом балансе прокси не возвращает явную ошибку, а даёт именно connection error. Откройте личный кабинет ProxyAPI и пополните счёт.
2. **OPENAI_BASE_URL** — точный URL из документации прокси (например `https://api.proxyapi.ru/v1`), без лишнего пути в конце.
3. **Доступ из контейнера** — с хоста прокси может открываться, из Docker — нет (другой DNS/сеть). Проверка с хоста: `curl -H "Authorization: Bearer YOUR_KEY" https://ваш-proxy/v1/models`.
4. **SSL** — если у прокси самоподписанный сертификат, в `.env` задайте `OPENAI_SSL_VERIFY=false`.
5. **Перезапуск** — после смены `.env`: `docker compose -p seo-agent up -d`.

**`project name must not be empty`**  
Запускайте команды из папки, где лежит `docker-compose.yml`, или явно задайте проект:  
`docker compose -p seo-agent up -d --build`
