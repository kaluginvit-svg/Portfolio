# Go для начинающих: сравнение с Python и разбор нашего проекта

Этот текст написан для тех, кто знаком с Python и почти не знаком с Go. Сначала — важная оговорка, затем — где Go уместен, и наконец — как устроен учебный сервер `userserver` в этом репозитории.

---

## 1. Go «лучше» Python? Не абсолютно

У языков нет единого рейтинга «лучше / хуже». У них разные компромиссы.

| Критерий | Go часто выигрывает | Python часто выигрывает |
|----------|---------------------|-------------------------|
| Скорость выполнения | Скомпилированный нативный код, быстрее на типичных CPU-задачах | Интерпретация + GIL ограничивают многопоточный CPU-код |
| Сборка и деплой | Один статический бинарник, без «виртуального окружения» на сервере | Нужен интерпретатор и зависимости (`pip`, venv) |
| Типы | Статическая проверка типов до запуска (меньше сюрпризов в проде) | Динамика удобна для быстрых скриптов и экспериментов |
| Экосистема под задачи | Сети, облако, infra, микросервисы — сильная стандартная библиотека | Data science, ML, автоматизация, прототипы — огромный PyPI |
| Обучение кривой | Явные типы, указатели, `error` — нужно привыкнуть | Старт проще: меньше формальной «церемонии» |

**Итог:** Go не «заменяет» Python вообще. Он часто выбирают, когда важны **предсказуемая производительность**, **простой деплой** и **один исполняемый файл**, а Python — когда важны **скорость разработки идей**, **научные библиотеки** и **гибкость типизации**.

---

## 2. Где Go обычно используют

- **Сетевые сервисы и API** (как наш HTTP-сервер): много готовых библиотек, хорошая работа с параллелизмом (goroutines).
- **CLI-утилиты** и **инфраструктурные инструменты** (Docker, Kubernetes, Terraform частично на Go).
- **Микросервисы** и **высоконагруженные бэкенды**, где важен контроль над памятью и задержками.
- **Бэкенды в облаке**, где один бинарник упрощает контейнеризацию.

Где чаще остаётся Python: **анализ данных**, **machine learning**, **скрипты**, **админка**, **Jupyter**, **быстрые MVP**, когда не критичны миллисекунды на запрос.

## 3. Как устроен наш проект на Go (как на карте)

Проект разбит на **пакеты** (`package`). Каждый каталог с `.go` файлами — отдельный пакет с одним именем.

- **`cmd/userserver`** — программа с точкой входа `func main()`. В Go исполняемый файл обычно живёт в `cmd/имя/`.
- **`internal/...`** — внутренние пакеты приложения: их не принято импортировать извне модуля (соглашение экосистемы Go).
  - **`config`** — настройки (пути к БД, параметры PBKDF2, адрес из переменных окружения).
  - **`models`** — структуры данных для JSON (поля запроса/ответа).
  - **`handlers`** — HTTP-слой: разбор запроса, коды ответов, вызов сервиса.
  - **`service`** — сценарий «создать пользователя» без привязки к HTTP.
  - **`repository`** — запись в SQLite и в файл паролей.
  - **`jsonutil`**, **`httputil`**, **`passwordhash`** — мелкие утилиты.

Такое разделение похоже на идею **слоёв** в Python: `handlers` ≈ Flask-маршруты, `service` ≈ `api.py`, `repository` ≈ репозитории.

## 4. Основы синтаксиса и идей Go — на примерах из проекта

### 4.1. Пакет и импорт

Первая строка файла — объявление пакета. У исполняемой программы это **`package main`**, и должна быть функция **`main()`**.

```go
package main

import (
    "database/sql"
    "userserver/internal/config"
)
```

- Импорты **группируют**: сначала стандартная библиотека, затем внешние модули, затем свой модуль (`userserver/...`).
- Пустой импорт `_ "modernc.org/sqlite"` подключает драйвер SQLite ради **побочного эффекта** (регистрация драйвера в `database/sql`), без прямого вызова в коде.

### 4.2. Объявление переменных и присваивание

Краткая форма:

```go
cfg := config.Default()
```

`:=` — объявить переменную и вывести тип из правой части.

Явные типы:

```go
var addr string
```

### 4.3. Ошибки явно, не исключения

В Go принято возвращать **`(результат, error)`**. Исключений `try/except` как в Python нет. Вызывающий код **обязан** обработать `err` (или явно проигнорировать, что на ревью обычно не нравится).

```go
db, err := sql.Open("sqlite", cfg.DBPath)
if err != nil {
    log.Fatalf("sqlite open: %v", err)
}
defer db.Close()
```

- **`defer`** — «выполни при выходе из функции» (здесь закрыть соединение). Порядок: `defer` выполняются в обратном порядку при return. Это удобно для освобождения ресурсов.

### 4.4. Структуры и «методы»

**Структура** — набор полей с типами (сродни `dataclass` в Python, но без магии по умолчанию).

```go
type UserService struct {
    Users    *repository.UserSQLite
    Password *repository.PasswordFile
}
```

**Метод** — функция с получателем `(s *UserService)`:

```go
func (s *UserService) CreateUser(name string, tags []string, password string) (int64, error) {
    // ...
}
```

- `*UserService` — **указатель** на структуру: метод может менять поля и не копирует всю структуру при каждом вызове.

### 4.5. Срезы (slice) и `append`

`[]string` — срез строк (динамический массив). Копирование и добавление тега `"new"` в сервисе делается так:

```go
rowTags := append(append([]string(nil), tags...), "new")
```

`...` — развернуть срез в аргументы. Это идиоматичный способ **скопировать** исходные теги и не портить переданный снаружи срез.

### 4.6. Форматирование ошибок

```go
return 0, fmt.Errorf("marshal tags: %w", err)
```

`%w` **оборачивает** ошибку — позже можно использовать `errors.Is` / `errors.As` для проверки причины.

### 4.7. HTTP-обработчик как замыкание

В `handlers` функция `newCreateUserHandler` возвращает **`http.HandlerFunc`** — функцию с сигнатурой `(ResponseWriter, *Request)`. Там же «захватывается» `svc` из внешней области — это похоже на замыкание в Python.

### 4.8. Константы из стандартной библиотеки

```go
http.MethodPost
http.StatusBadRequest
```

Так вместо «магических строк» везде используются именованные константы.

## 5. Как это соотносится с Python-версией проекта

| Идея | Python | Go (этот репозиторий) |
|------|--------|------------------------|
| Точка входа | `python app.py` или `flask run` | `go run ./cmd/userserver` или собранный `.exe` |
| Зависимости | `requirements.txt`, venv | `go.mod` + кэш модулей; бинарник один |
| Конфигурация | `settings.py`, `os.environ` | `internal/config`, переменные окружения |
| HTTP | Flask-маршруты | `handlers` + `net/http` |
| Слой «сценария» | `api.py` | `service` |
| БД и файлы | `users_repository`, `password_storage` | `repository` |

Поведение API **намеренно** похоже: те же пути, коды ответов и формат строки с PBKDF2.

## 7. Краткое резюме

- **Go** — компилируемый язык с **статической типизацией**, явными **ошибками**, **простым деплоем** одного бинарника.
- **Python** — быстрее для многих экспериментов и доменов науки/скриптов; **Go** — часто удобнее для **долгоживущих сервисов** и **инфраструктурных утилит**.
- В этом проекте Go показан **как набор пакетов**: `main` → `handlers` → `service` → `repository`, плюс утилиты и модели — это хорошая отправная точка, чтобы читать код и `go doc` по мере необходимости.
