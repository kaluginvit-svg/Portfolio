# Архитектура системы управления проектами

Этот документ описывает базовую архитектуру для нашей системы управления задачами. Цель — создать гибкую и масштабируемую основу.

Мы будем придерживаться подхода с **Сервисным слоем**, где модели данных (`Task`, `Project`) остаются простыми, а вся бизнес-логика выносится в отдельные сервисы (`TaskService`, `ProjectService`).

---

## 1. Основные классы (Модели данных)

Нам понадобятся четыре ключевых класса для хранения данных.

### `User` (Пользователь)

Представляет аккаунт пользователя в системе.

*   **Атрибуты:**
    *   `id` (UUID): Уникальный идентификатор.
    *   `username` (string): Имя пользователя для входа и отображения.
    *   `email` (string): Электронная почта, уникальна.
    *   `created_at` (datetime): Дата и время регистрации.

*   **Методы (базовые):**
    *   `update_profile(new_username)`: Изменяет данные профиля.

### `Project` (Проект)

Логическая группа для объединения задач.

*   **Атрибуты:**
    *   `id` (UUID): Уникальный идентификатор.
    *   `name` (string): Название проекта.
    *   `description` (text): Описание проекта.
    *   `owner_id` (FK -> User.id): ID пользователя, который создал и владеет проектом.
    *   `created_at` (datetime): Дата создания.

*   **Методы (базовые):**
    *   `archive()`: Архивирует проект, делая его недоступным для изменений.
    *   `change_name(new_name)`: Меняет название проекта.

### `Task` (Задача)

Конкретная единица работы внутри проекта.

*   **Атрибуты:**
    *   `id` (UUID): Уникальный идентификатор.
    *   `title` (string): Краткое название задачи.
    *   `description` (text, nullable): Подробное описание.
    *   `status` (Enum): Статус задачи (`'ToDo'`, `'InProgress'`, `'Done'`).
    *   `due_date` (date, nullable): Срок выполнения.
    *   `project_id` (FK -> Project.id): ID проекта, к которому относится задача.
    *   `assignee_id` (FK -> User.id, nullable): ID пользователя, назначенного ответственным.
    *   `created_at` (datetime): Дата создания.

*   **Методы (базовые):**
    *   `mark_as_done()`: Устанавливает статус `'Done'`.
    *   `reopen()`: Устанавливает статус `'ToDo'`.
    *   `set_due_date(new_date)`: Изменяет срок выполнения.

### `Membership` (Членство в проекте)

Связующая модель, которая определяет, какой пользователь состоит в каком проекте и с какой ролью. Это ядро нашей системы прав доступа.

*   **Атрибуты:**
    *   `user_id` (FK -> User.id): ID участника.
    *   `project_id` (FK -> Project.id): ID проекта.
    *   `role` (Enum): Роль в проекте (`'Admin'`, `'Member'`, `'Guest'`).

*   **Методы:**
    *   Нет. Это чистая data-class модель для связи.

## 2. Взаимодействие объектов

Связи между объектами строятся следующим образом:

1.  **Пользователь и Проект (Многие-ко-многим):**
    *   Один `User` может состоять во многих `Project`.
    *   Один `Project` может иметь много `User`.
    *   Эта связь реализуется через промежуточную модель `Membership`. **Запрещено** создавать прямую связь "многие-ко-многим" между User и Project.

2.  **Проект и Задача (Один-ко-многим):**
    *   Один `Project` содержит в себе список (коллекцию) объектов `Task`.
    *   Каждая `Task` принадлежит **только одному** `Project`.

3.  **Пользователь и Задача (Один-ко-многим):**
    *   Один `User` может быть назначен ответственным (`assignee`) за множество `Task`.
    *   У каждой `Task` может быть **только один** `assignee` (или ни одного).

## 3. Логика и Действия (Сервисный слой)

Вся сложная бизнес-логика, требующая взаимодействия нескольких объектов и проверки прав, выносится в сервисы. Модели данных остаются "глупыми".

### `ProjectService`
*   **Отвечает за:** создание проектов, управление участниками и правами.
*   **Примеры методов:**
    *   `create_project(user: User, name: string)`: Создает проект, назначает `user` его владельцем (`owner`) и добавляет его в `Membership` с ролью `'Admin'`.
    *   `add_user_to_project(acting_user: User, project: Project, target_user: User, role: string)`: Добавляет `target_user` в проект. **Внутри сервис проверяет, имеет ли `acting_user` права администратора в этом проекте.**
    *   `get_project_tasks(user: User, project: Project)`: Возвращает список задач проекта. **Внутри сервис проверяет, имеет ли `user` доступ к этому проекту.**

### `TaskService`
*   **Отвечает за:** управление жизненным циклом задач.
*   **Примеры методов:**
    *   `create_task(user: User, project: Project, title: string)`: Создает задачу. **Проверяет, является ли `user` участником проекта.**
    *   `assign_task(acting_user: User, task: Task, target_user: User)`: Назначает `target_user` на задачу. **Проверяет, что `acting_user` имеет права на это действие, и что `target_user` является участником проекта.**
