# Манифест проектирования и взаимодействия с ИИ

Этот документ описывает опыт генерации кода по архитектурному файлу и выводы, сделанные в процессе.

---

## 1. Исходный промт

```
Привет! Я проектирую систему управления задачами на Python с использованием ООП. 
Вот архитектура на русском языке: [СОДЕРЖАНИЕ architecture.md]. 

Пожалуйста, напиши полный код на Python, реализующий эту архитектуру. 
Создай классы, которые я описал, с атрибутами и методами. 
В конце файла добавь пример создания нескольких объектов и вызова их методов, 
чтобы продемонстрировать работу системы.
```

---

## 2. Что получил

**Один файл**, содержащий в себе все:
- Все классы моделей (`User`, `Project`, `Task`, `Membership`)
- Все сервисы (`ProjectService`, `TaskService`)
- Примеры использования

---

## 3. Что исправил

Изначальный результат был неоптимален для реального проекта. Пришлось отправлять дополнительные уточняющие промты.

### Промт-исправление:

```
Раздели код на отдельные файлы по принципу "один файл — один объект". 
Также создай одноименные файлы для каждого сервиса.
```

### Результат исправления:

Получил правильную структуру проекта:
- Каждый класс в отдельном файле
- Каждый сервис в отдельном файле
- Корректные импорты между модулями

### Замечания:

Пропуски методов или аргументов не заметил — ИИ справился с этой частью хорошо.

---

## 4. Вывод

**Главный урок: надо все делать сразу одним, намного более обширным промптом.**

### Что нужно было сделать иначе:

Вместо итеративного подхода (запрос → исправление → доработка), следовало **сразу** в первом промте указать:

1. **Архитектуру системы** (какие классы, атрибуты, методы)
2. **Структуру файлов** (один файл — один класс/сервис)
3. **Организацию папок** (модели в одной папке, сервисы в другой)
4. **Требования к импортам** (все зависимости должны быть корректными)
5. **Примеры использования** (демонстрация работы системы)

### Улучшенная версия промта:

```
Привет! Я проектирую систему управления задачами на Python с использованием ООП.

Вот архитектура: [СОДЕРЖАНИЕ architecture.md]

Требования к реализации:
1. Создай структуру проекта с папками:
   - src/models/ — для классов моделей
   - src/services/ — для сервисов
   - main.py — для примеров использования

2. Каждый класс модели в отдельном файле:
   - src/models/user.py
   - src/models/project.py
   - src/models/task.py
   - src/models/membership.py

3. Каждый сервис в отдельном файле:
   - src/services/project_service.py
   - src/services/task_service.py

4. Создай файл src/models/__init__.py для удобного импорта моделей
5. Создай файл src/services/__init__.py для удобного импорта сервисов
6. Убедись, что все импорты корректны и код запускается
7. В main.py добавь примеры использования системы

Реализуй эту архитектуру полностью.
```

---

## 5. Практические выводы

### ✅ Что работает хорошо:

- Четкая архитектура в отдельном файле (architecture.md)
- ИИ хорошо понимает структуру классов и их взаимосвязи
- Методы и атрибуты генерируются корректно

### ⚠️ Что требует внимания:

- **Структура файлов** — ИИ не угадывает это автоматически, нужно явно указывать
- **Организация папок** — по умолчанию ИИ создаст один большой файл
- **Импорты** — при разделении на файлы нужно проверять зависимости

### 📝 Рекомендации:

1. **Инвестируй время в первый промт** — подробное описание требований экономит время на исправлениях
2. **Указывай физическую архитектуру** (папки, файлы), а не только логическую (классы, методы)
3. **Проси явно про импорты** — это частая проблема при разделении кода
4. **Тестируй сразу** — проверяй, что код запускается и работает
